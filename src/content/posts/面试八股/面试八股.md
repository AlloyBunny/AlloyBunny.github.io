# RL篇

## RL怎么调参

### 有哪些参数

1. **资源参数**：决定显存、吞吐，通常尽量开到硬件可承受。
   - Micro batch size（并行跑多少个forward/backward，开大会更快）
   - 最大序列长度（如果95+%的样本长度都在3k以内，少数到7k且不是重要样本，那应该设置3k）
   - Rollout 并发数（rollout_concurrency，开大会更快）
2. **更新强度参数**：决定每轮把模型推多远，是最容易训炸的部分。
   - learning rate（SFT可以从1e-4开始，RL可以从1e-6开始）
   - Critic learning rate（PPO特有。Critic的学习率是Actor的多少倍，一般2～10倍）
   - KL散度的$\beta$
   - 裁剪的$\epsilon$
3. **采样与目标参数**：决定训练信号质量，不是越大越好，需要看 reward、KL、clip fraction 等曲线。
   - train batch size（每个 step 采样多少个 prompt，也叫prompt batch size或者rollout batch size）
   - group size（同一个 prompt 采样 $G$ 条回复）
   - max length（分为max_prompt_length和max_response_length，后者很影响显存消耗）
   - PPO epochs（每个采样轨迹用来更新的次数）



### 调参优先级

1. Reward 的好坏和数据难度，比调参更重要
2. **OOM调参：**降低prompt_batch_size、降低rollout并发数、降低最大序列长度、降低micro batch size、降低group size
3. **LR：**可以先把1e-6 / 2e-6 / 5e-6都试一遍，然后看情况调（见下节）
4. **group size：**在verl里，rollout.n就是group size，学不动的时候需要增大它
5. **PPO epochs：**每个采样轨迹用来更新的次数。作用类似LR，更新不足的时候增大它，更新过度的时候降低它
6. **KL $\beta$, clip $\epsilon$：**KL 持续增长、输出越来越长、语言/格式退化、eval能力下降的时候，增大KL（约束收紧）

### 根据现象调参

1. **OOM了：**降低rollout并发数、降低最大序列长度、降低group size
2. **reward 不涨**：增大LR、增大group size、降低KL、检查是否出现大量全对或全错
3. **reward上涨但模型崩了：**减小LR、增大KL $\beta$、减小clip $\epsilon$、降低PPO epochs
4. **reward 快速上涨，但 eval 不涨：**如果是reward hacking要调reward、如果是过拟合要降低PPO epochs、降低LR、增大KL $\beta$、减小clip $\epsilon$
5. **GRPO大量全对/全错：**增大group size、增大温度、修改数据采样
6. **KL 突然爆炸：**减小LR、增大KL $\beta$、减小clip $\epsilon$、减小PPO epochs
7. **裁剪比例过高：**减小LR、减小PPO epochs
8. **输出太长：**加入长度惩罚、增大KL $\beta$、减小clip $\epsilon$、减小max length
9. **reward震动：**降低LR、增大group size、增大KL $\beta$、减小clip $\epsilon$

## RL的流程

流程如下：

```
采样（Rollout）
      ↓
计算奖励和Advantage
      ↓
策略更新（Update）
      ↓
重新采样
      ↓
...
```

### 多久交替一次？

**由rollout batch size决定**，举个例子，用GRPO，数据集有10k个prompts，rollout batch = 512，G=8。那么每次选中512个prompts，每个生成8个response，生成总共4096个trajectories，然后拿去计算奖励，策略更新。

### 策略更新的时候Forward/Backward在干嘛？

forward是把prompt + response喂给模型，让它算一遍$\log\pi_\theta(y|x)$，然后和rollout时期已经计算并保存了的$\log\pi_{\mathrm{old}}(y|x)$结合起来用来算重要性采样系数$r_t(\theta)=\frac{\pi_\theta(y_t|x,y_{<t})}{\pi_{\mathrm{old}}(y_t|x,y_{<t})}$

### GRPO实际训练循环可以理解成：

```python
for _ in training_steps:
	# =================
	# Rollout
	# =================
	prompts = sample_prompts(train_batch_size) # 一次采样prompt_batch_size个prompts
	responses = policy.generate(prompts,G=group_size) # 根据x采样出y，每个x采样group_size个y。
	old_logp = old_policy.logprob(responses) # 计算log π_old(y|x)
	rewards = evaluator(responses)
	advantages = compute_advantage(rewards,G=group_size) # 计算组内相对优势

	# =================
	# Update
	# =================
	for epoch in range(GRPO_epochs): # 一批采样会经过GRPO_epochs次更新，因为采样贵，更新快
		for minibatch in rollout_data:
			optimizer.zero_grad()
			for microbatch in split(minibatch): # 同一个minibatch内是并行的
				new_logp = policy.logprob(microbatch.responses) # forward，计算log π_θ(y|x)
				ratio = exp(new_logp - microbatch.old_logp) # 重要性采样系数
				loss = compute_loss(ratio, microbatch.advantages)
				backward()
			optimizer.step() # 更新参数
```



# LLM推理篇

## vLLM怎么进行推理加速

### PageAttention（类似OS中的分页）

**问题：**并行推理的时候，传统注意力计算需给KV cache预留最大序列长度内存，导致显存浪费（如用户输入短文本时，80%显存闲置）

**方法：**PageAttention按照类似OS中分页技术的概念，**把KV cache划分为多个KV blocks**，按块管理，每个sequence占的块的显存地址不需要连续，减少显存碎片。

PageAttention也有block table，类似分页技术中的页表。逻辑KV cache是连续的，而实际的物理KV cache不连续，block table负责把逻辑KV cache地址映射到物理KV cache。

### Continuous Batching（连续批处理）

**问题：**静态批处理需等待所有请求完成后统一计算，GPU利用率低

```
batch 1:

request A
request B
request C

全部结束
↓
batch 2
```

**方法：**Continuous Batching实时监控请求状态，动态合并正在处理的请求（类似CPU流水线）：

- 迭代级调度：每生成一个token重新调整批次（如果有request跑完了，空位会被立刻补上）
- 抢占式推理：优先处理高优先级的请求；把占用太多显存的请求先swap到CPU memory

```
（以下每个step代表4个request一起生成next token）
step1:
A B C D
step2:
A B C D
step3:
E B C D
step4:
E B F D
```

### Sharing KV Blocks（共享KV Block）

**问题：**在用同一个prompt进行多次推理，或者进行bean search时，不同request用到的KV cache有很长一段时重叠的，能否把这些地方的KV block只存一次？

**方法：**重复的KV block只计算、缓存一份

# 显存优化篇

## 模型SFT显存估计

**重要原则：**现在模型默认用bf16/bf16精度，16位是2 bytes，$1\ \text{B} * 2\ \text{bytes} \approx 2\ \text{GB}$。即每1B的16位对应2GB。注意，之所以这里是约等于，是因为模型参数量的7B是7e9，不是7×2^30，所以准确的计算方法是$1\ \text{B} * 2\ \text{bytes}=2*\frac{10^9}{2^30}\text{GB}$，略小于2GB。

**静态开销：**全参量SFT，假设模型参数是ψ。模型参数是2ψ，梯度是2ψ，master weights是4ψ，一阶矩和二阶矩各4ψ（注：master weights、一阶矩、二阶矩都是float 32），总共是16ψ。所以7B全参量SFT至少需要16×7=112GB。而如果是LoRA的SFT，假设模型参数是ψ，LoRA的总参数量是φ，则静态开销为ψ+16φ≈ψ，所以LoRA能省很多显存。

**动态开销：**英伟达给了个近似公式，如下：
$$
\text{Mem\_Act}=\mathrm{L}\cdot\mathrm{s}\cdot\mathrm{b}\cdot\mathrm{h}\times(34+5\cdot\mathrm{a}\cdot\mathrm{s}/\mathrm{h})\mathrm{bytes}
$$
其中，L是模型层数，s是sequence length，b是micro-batch size，h是hidden size，a是attention heads。面试中不用死记这个公式，只需要记住：$\text{Mem\_Act} \propto Lbs^2$

**LoRA省了哪些：**LoRA省了静态开销中除了模型参数外的部分，几乎不省动态开销

## 模型推理显存估计

推理显存=模型权重+KV Cache，其中KV Cache的占用计算公式为：
$$
KV\approx 2\times L\times H\times d_{head}\times seq\times batch\times dtype
$$
其中，各参数分别是：

| 参数       | 含义                                 | 例子（Llama/Qwen）       |
| ---------- | ------------------------------------ | ------------------------ |
| $L$        | Transformer 层数（number of layers） | Llama-2-7B: 32           |
| $H$        | Attention head 数量                  | Llama-2-7B: 32           |
| $d_{head}$ | 每个 head 的维度                     | 128                      |
| $seq$      | 当前上下文长度（sequence length）    | 4096                     |
| $batch$    | batch size（同时推理多少条）         | 8                        |
| $dtype$    | KV Cache 数据类型大小                | FP16=2 bytes, FP8=1 byte |

## 节省显存的技术

1. 梯度累加：把多个micro-batch累加起来得到一个batch/mini-batch的累计梯度，不用真的把micro-batch开大
2. 激活重算：时间换空间，丢弃部分前向激活值，在反向传播时重新计算
3. ZeRO/FSDP：将参数、梯度和优化器状态切分到多张GPU，训练时间换显存节省
4. FlashAttention：硬件层优化自注意力内存读写，避免显式保存注意力分数矩阵

## 常见的几种并行技术

```
Parallelism
│
├── Data Parallelism
│   ├── DDP（每个GPU放一个完整模型，只加速，不省显存）
│   └── ZeRO / FSDP（能省显存，其中FSDP相当于pytorch版ZeRO-3）
│
├── Model Parallelism
│   ├── Tensor Parallelism（一个权重矩阵W拆成几块给多个GPU。hidden_size维度拆）
│   └── Pipeline Parallelism（不同layer拆给不同GPU。layer_num维度拆）
│
├── Sequence Parallelism（把上下文在sequence_length维度拆，省activation显存，即节省动态开销）
│
└── Expert Parallelism (MoE)
```

其中，数据并行：

|        | 参数切分 | 梯度切分 | 优化器切分 |
| ------ | -------- | -------- | ---------- |
| DDP    | ❌        | ❌        | ❌          |
| ZeRO-1 | ❌        | ❌        | ✅          |
| ZeRO-2 | ❌        | ✅        | ✅          |
| ZeRO-3 | ✅        | ✅        | ✅          |
| FSDP   | ✅        | ✅        | ✅          |

## Flash Attention

**问题：**$S = QK^T$的$S$矩阵的大小是$s*s$，太大了，写入显存(HBM)太慢。

**方法：**SRAM写入速度很快，所以可以在SRAM上分成多个小块计算$S = QK^T$，不用写入显存，就能快很多

**作用：**1. 计算更快（主要）2. 显著降低注意力机制的激活峰值（还能省显存）

**注释：**Flash Attention是个硬件层面的优化，不改变Attention的数学计算原理
